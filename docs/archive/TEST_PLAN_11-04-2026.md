## Jak używać tego pliku

**Nie testuj tego zestawu na kodzie z commita `bc765a2`** (stan przed 11.04 popołudnie). Kod był pisany przeciwko wcześniejszemu specowi — większość testów z tej listy wywali się jako FAIL nie dlatego że coś jest złamane, tylko dlatego że nowy spec jeszcze nie jest zaimplementowany. To byłoby dokumentowanie driftu który już jest udokumentowany.

**Używaj tego zestawu dopiero po:**

1. Przejściu audytu kodu → `docs/CODE_AUDIT_11-04-2026.md` (tabela rozbieżności)
2. Triage z Maanem: must / should / nice priorytety
3. Fixach must-have (per Rule #11 — jeden na raz, commit per fix)

Każdy fix rozbrojoje podzbiór testów z tej listy — np. fix "3-button cards w confirmation keyboard" rozbroi testy 1, 2, 3, 6, 7, 9, 10, 11. Test weryfikuje że fix faktycznie działa, nie że kod w ogóle istnieje.

**Per Rule #11, dla każdego przeprowadzonego testu zapisz dowód:**
- (a) Screenshot karty z Telegrama przed zapisem
- (b) Screenshot potwierdzenia po `✅ Zapisać`
- (c) Screenshot z Google Calendar pokazujący event (dla testów z Calendarem)
- (d) Screenshot z Google Sheets pokazujący wiersz (dla testów z Sheets)
- (e) Wynik (PASS / FAIL / PARTIAL + opis) zapisać do `CURRENT_STATUS.md` sekcja "Historia sesji"

## Test 1 — Proste spotkanie (emoji 📅)

**YOU SEND:** `Spotkanie z Janem Kowalskim z Warszawy jutro o 14`

**BOT SHOULD REPLY:**
- Karta z 📅 w nagłówku
- Pełne imię + nazwisko + miasto: "Jan Kowalski, Warszawa"
- Data w formacie `DD.MM.YYYY (Dzień tygodnia) 14:00` (nie ISO, nie raw)
- Trzy przyciski: `[✅ Zapisać]` `[➕ Dopisać]` `[❌ Anulować]`

**Po `✅ Zapisać`:**
- Event w Google Calendar na jutro 14:00 zawierający "Jan Kowalski" i "Warszawa" w tytule albo opisie
- Bot pyta R7 otwartym tekstem typu: "Co dalej z Janem Kowalskim? Spotkanie, telefon, follow-up?"

**FAIL IF:**
- Karta ma stary wzorzec `[Tak][Nie]` albo `[Zapisz bez]` zamiast 3-button
- Karta ma 📞 albo 📨 zamiast 📅 (user wyraźnie napisał "spotkanie")
- Event w Calendarzu nie powstał mimo zapisu
- Bot po zapisie pokazuje sztywną trójkę `[Spotkanie][Telefon][Nic]` zamiast free-text R7

## Test 2 — Telefon (emoji 📞)

**YOU SEND:** `Zadzwonić do Anny Nowak z Krakowa w piątek o 10`

**BOT SHOULD REPLY:**
- Karta z 📞 w nagłówku (nie 📅)
- Tytuł typu "Telefon: Anna Nowak" albo wskazanie że to telefon nie spotkanie
- Data w formacie `DD.MM.YYYY (Dzień tygodnia) 10:00` (najbliższy piątek)
- Trzy przyciski 3-button

**Po `✅ Zapisać`:**
- Event w Calendar w najbliższy piątek 10:00
- Tytuł eventu wskazuje że to telefon (prefix "Telefon:" albo analogiczny)

**FAIL IF:**
- Bot rozpoznaje to jako spotkanie 📅 (nie rozróżnia rodzaju kontaktu)
- Tytuł eventu w Calendarzu nie różni się od zwykłego spotkania

## Test 4 — Polski czas

**YOU SEND:** `Spotkanie z Marią Lewandowską z Poznania jutro wpół do ósmej rano`

**BOT SHOULD REPLY:**
- Karta pokazuje `07:30`
- Nie `07:00`, nie `08:00`, nie `19:30` (wieczór)

**Po `✅ Zapisać`:**
- Event w Calendar jutro o 07:30

- Event w wieczornym slocie (19:30) mimo "rano"

## Test 5 — Data względna "za tydzień"

**YOU SEND:** `Spotkanie z Tomaszem Kowalczykiem z Łodzi za tydzień o 16`

**BOT SHOULD REPLY:**
- Karta pokazuje datę dokładnie 7 dni od dzisiaj
- Dzień tygodnia poprawnie obliczony (np. `18.04.2026 (Sobota) 16:00`)

**Po `✅ Zapisać`:**
- Event w Calendar dokładnie za 7 dni o 16:00

**FAIL IF:**
- Data jest inna niż +7 dni (np. "w przyszły tydzień" interpretowane jako poniedziałek)
- Dzień tygodnia źle obliczony

## Test 6 — One-click cancel

**YOU SEND:** Dowolne spotkanie z testów 1-5

**Na karcie kliknij:** `❌ Anulować`

**BOT SHOULD REPLY:**
- Natychmiastowe potwierdzenie typu "Anulowano."
- **NIE** pyta `Na pewno anulować?`
- **NIE** wymaga drugiego kliknięcia

**Weryfikacja:**
- Żadnego eventu w Calendarzu
- Żadnego wiersza w Sheets (jeśli był compound)
- Pending state czysty — napisz `cześć` i sprawdź że bot odpowiada normalnie, nie próbuje dokończyć poprzedniego flow

**FAIL IF:**
- Pętla potwierdzenia `Na pewno anulować?` → `Tak/Nie`
- Pending state zostaje aktywny — następna wiadomość jest interpretowana jako dokończenie poprzedniego flow

## Test 7 — `➕ Dopisać` rebuild karty

**YOU SEND:** `Spotkanie z Janem Kowalskim z Warszawy jutro o 14`

**Na karcie kliknij:** `➕ Dopisać`

**Potem napisz:** `w domu klienta, adres Marszałkowska 100`

**BOT SHOULD REPLY:**
- **Ta sama karta**, przebudowana (nie nowa karta obok)
- Zawiera dodane miejsce/adres
- Nadal trzy przyciski `[✅ Zapisać] [➕ Dopisać] [❌ Anulować]`

**Po `✅ Zapisać`:**
- Event w Calendar ma uzupełnioną lokalizację / opis z adresem

**FAIL IF:**
- Bot utworzył dwa osobne eventy albo dwie karty
- Adres nie został dołączony do pending
- `➕ Dopisać` zamknął flow zamiast go otworzyć

## Test 8 — Kolizja terminu

**Przygotowanie:** Najpierw wykonaj test 1 (spotkanie na jutro 14:00 zapisane).

**YOU SEND:** `Spotkanie z Ewą Zielińską z Gliwic jutro o 14`

**BOT SHOULD REPLY:**
- Karta z ostrzeżeniem o kolizji z wcześniejszym eventem (Jan Kowalski)
- Nadal trzy przyciski (user decyduje sam czy mimo kolizji)
- **NIE** blokuje automatycznie, tylko informuje

**Po `✅ Zapisać` mimo kolizji:**
- W Calendarzu **dwa** eventy jutro 14:00 (Jan Kowalski + Ewa Zielińska)

**FAIL IF:**
- Bot nie wykrył kolizji (brak ostrzeżenia)
- Bot zablokował zapis zamiast zostawić decyzję userowi

## Test 9 — `add_client + add_meeting` (compound fusion)

**YOU SEND:** `Nowy klient Adam Nowicki z Wrocławia, PV 10kW, tel 600100200, spotkanie jutro o 15`

**BOT SHOULD REPLY:**
- **JEDNA** karta compound z sekcjami "📋 Do Sheets" i "📅 Do Kalendarza" (nie dwie osobne)
- Moc `10kW` widoczna w sekcji Notatki (nie w nazwie produktu)
- Produkt = `PV` (bez "10kW" w nazwie)
- Trzy przyciski 3-button

**Po `✅ Zapisać`:**
- Nowy wiersz w Sheets: Adam Nowicki, Wrocław, PV, tel 600100200, Notatki zawierają `10kW`
- Event w Calendar jutro 15:00
- R7 pyta otwarcie "Co dalej z Adamem?"

**FAIL IF:**
- Dwie osobne karty zamiast jednej
- Moc `10kW` w kolumnie `Produkt` zamiast `Notatki`
- Brak eventu w Calendarzu albo brak wiersza w Sheets (atomicity failed)

## Test 12 — Duplicate detection w compound

**Przygotowanie:** Test 1 już zapisany → w Sheets jest Jan Kowalski z Warszawy.

**YOU SEND:** `Nowy klient Jan Kowalski z Warszawy, tel 600100200, spotkanie w środę o 11`

**BOT SHOULD REPLY:**
- Bot wykrywa duplikat (Jan + Kowalski + Warszawa match)
- **2-przyciskowa** karta disambiguation: `[📋 Dopisz do istniejącego] [➕ Utwórz nowy wpis]`
- **NIE** pokazuje starego `[Nowy][Aktualizuj]`

**Kliknij:** `📋 Dopisz do istniejącego`

**BOT SHOULD REPLY:**
- Karta compound pokazująca update istniejącego Jana + nowe spotkanie
- Trzy przyciski 3-button

**Po `✅ Zapisać`:**
- Ten sam wiersz Jana ma zaktualizowany telefon (jeśli był inny) albo zostaje
- Event w Calendar w środę 11:00
- **NIE** powstał nowy wiersz Jana (nadal tylko jeden)

**FAIL IF:**
- Stary `[Nowy][Aktualizuj]` keyboard
- Bot utworzył drugiego Jana Kowalskiego mimo kliknięcia "Dopisz"
- Brak eventu w Calendarzu

## Kolejność wykonywania testów — praktyka

**Przed testami 1-8:** arkusz może być pusty (te testy nie piszą do Sheets, tylko read + Calendar).

**Przed testami 9-12:** wyczyść arkusz albo ustaw znany stan. Po każdym compound teście zanotuj który wiersz się dopisał.

**Zależności między testami:**
- Test 8 wymaga że test 1 już został zapisany (istniejący event jutro 14:00 dla kolizji).
- Test 12 wymaga że test 1 już zapisał Jana Kowalskiego w Sheets (dla duplicate detection).
- Test 13 i 14 najlepiej na różnych klientach (żeby nie mieszać R7 state).

**Po każdym teście** wpisz wynik do `docs/CURRENT_STATUS.md` sekcja "Historia sesji" w formacie:

## Co ten plan NIE pokrywa

**Świadomie poza scope tej listy (do ewentualnych osobnych testów):**

- **Voice input (Faza 5)** — nagrania głosowe, Whisper STT, low-confidence disambiguation. Nie dotykamy w pierwszej rundzie testów, Faza 5 jest `⏳ TODO`.
- **Drive / zdjęcia (Faza 4)** — wysyłanie zdjęć, Google Drive upload. Faza 4 jest `⏳ TODO`.
- **Morning brief / proactive messages (Faza 6)** — automatyczne wiadomości poranne i wieczorne. Nie ma ich w kodzie.
- **Lejek sprzedażowy** — POST-MVP banner, nie pełna implementacja.
- **POST-MVP intencje:** `edit_client` pełna edycja, `filtruj_klientów`. Gdy wejdą, wymagają własnych testów.
- **NIEPLANOWANE intencje:** `reschedule_meeting`, `cancel_meeting`, `free_slots`, `meeting_non_working_day_warning`. **Pozytywny test** dla nich: user wpisuje "przesuń spotkanie z Janem na jutro" → bot powinien odpowiedzieć "Reschedule nie jest w MVP — przesuń ręcznie w Google Calendar, bot nie ruszy". To może być **test 16** jeśli chcemy explicite weryfikować że wycięcia są egzekwowane.

## Smoke test (1 minuta, można zrobić przed audytem)

Jedyny test który ma sens PRZED audytem kodu — weryfikacja że bot w ogóle się uruchamia:

**YOU SEND:** `cześć`

**BOT SHOULD REPLY:** Jakąkolwiek sensowną odpowiedź powitalną albo prośbę o doprecyzowanie.

**FAIL IF:** Bot nie odpowiada, crashuje, error 500, "internal server error".

Ten test zajmuje 30 sekund i weryfikuje że zmiana branchy / deployu nie zepsuła infra. Nic więcej nie testuj dopóki audyt + fixy nie są zrobione.
